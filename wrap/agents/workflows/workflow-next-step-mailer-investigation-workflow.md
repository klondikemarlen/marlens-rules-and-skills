# Workflow Next Step Mailer Investigation Workflow

## Intent

**WHY this workflow exists:** Next-step notification investigations can become slow when agents search broadly across `email_histories`, workflow access, history, delegations, and queued mailer jobs without first identifying the specific workflow, step order, and relevant users. This workflow narrows the investigation quickly.

**WHAT this workflow produces:** A concise evidence trail and a shareable narrative report showing:

- Which `NextStepMailer` jobs were queued for the workflow and linked step order
- Which workflow step id the email hash referred to at send time and now
- Which users or positions were step players
- Which users had workflow or step access, and why
- Which workflow history events explain sends, send-backs, reordering, or player changes
- Whether delegation was relevant on the dates in question
- A plain-language timeline suitable for an arbitrary workflow, step hash, and set of players

## Decision Rules

- **Use `background_jobs` for `NextStepMailer`:** `NextStepMailer` jobs are the primary evidence source. Do not start from `email_histories` for “You’re Up Next” emails.
- **Treat `email_histories` as supplemental only:** If it is empty for this mailer, that is expected and not evidence that no email was sent.
- **Interpret hash fragments as step order:** A link like `#step-10` means `workflow_steps.[order] = 10`, not `workflow_steps.id = 10`.
- **Do not search for the hash in job JSON:** The hash is generated by the email template from the serialized step order. `background_jobs.job_data` stores the workflow and step data, not the literal `#step-<order>` fragment.
- **Use staged queries:** First identify user ids, workflow id, step order, and job rows. Then use those ids in focused follow-up queries.
- **Include historical rows when reconstructing events:** For step players and access rows, deleted rows often explain what happened at send time.
- **Separate working evidence from shareable output:** During investigation, paste concise result rows below queries if useful. Before sharing, remove raw query output, comments, emails, and any unnecessary private data.
- **Do not force one mega-query:** It is acceptable to ask the user to run one query, paste ids/results, then run the next focused query.
- **Prefer role-based wording in final reports:** Use personal names only when needed to explain actions, access, or reported wording. Prefer role labels such as `workflow creator`, `review step player`, `delegatee`, or `position holder` when names are not necessary.

## Inputs

Ask the user for as many of these as they know:

- **Workflow slug:** Example `<workflow-slug>`
- **Reported email link or hash:** Example `/workflows/<workflow-slug>/steps#step-<order>`
- **Reported wording:** Example `Please perform <Action> to unblock <Next Action> for <Player>`
- **Named users:** Names or emails from the report
- **Approximate date range:** When the email was received or expected

If the user only has a link, derive:

- `workflow_slug` from the URL path
- `linked_workflow_step_order` from `#step-<number>`

## Step 1: Resolve Users

Use names and emails from the report. Keep this first query intentionally broad. The user can then provide the exact ids for later queries.

```sql
SELECT
  users.id,
  users.email,
  users.display_name,
  users.first_name,
  users.last_name,
  users.is_active,
  users.email_notifications_enabled,
  users.deleted_at
FROM
  users
WHERE
  LOWER(users.email) IN (
    '<email-1>',
    '<email-2>'
  )
  OR LOWER(users.display_name) IN (
    '<display-name-1>',
    '<display-name-2>'
  )
  OR LOWER(CONCAT(users.first_name, ' ', users.last_name)) IN (
    '<display-name-1>',
    '<display-name-2>'
  )
ORDER BY
  users.display_name,
  users.email,
  users.id;
```

Record:

- **Investigation user ids**
- **Whether each user is active**
- **Whether email notifications are enabled**

## Step 3: Trace `NextStepMailer` Jobs for the Linked Step Order

This is the most important query for “You’re Up Next” email reports.

```sql
SELECT
  background_jobs.id AS background_job_id,
  background_jobs.created_at AS job_created_at,
  background_jobs.enqueued_at AS job_enqueued_at,
  background_jobs.status,
  background_jobs.queue_name,
  background_jobs.job_name,
  background_jobs.job_filename,
  JSON_VALUE(background_jobs.job_data, '$[0].slug') AS workflow_slug,
  JSON_VALUE(background_jobs.job_data, '$[0].title') AS workflow_title,
  TRY_CONVERT(int, JSON_VALUE(background_jobs.job_data, '$[1].id')) AS workflow_step_id,
  TRY_CONVERT(int, JSON_VALUE(background_jobs.job_data, '$[1].order')) AS workflow_step_order,
  JSON_VALUE(background_jobs.job_data, '$[1].title') AS workflow_step_title,
  JSON_VALUE(background_jobs.job_data, '$[1].performerId') AS workflow_step_performer_id,
  JSON_VALUE(background_jobs.job_data, '$[1].actualStartDate') AS workflow_step_actual_start_date,
  background_jobs.job_data
FROM
  background_jobs
WHERE
  background_jobs.queue_name = 'mailer'
  AND background_jobs.job_name = 'NextStepMailer'
  AND JSON_VALUE(background_jobs.job_data, '$[0].slug') = '<workflow-slug>'
  AND TRY_CONVERT(int, JSON_VALUE(background_jobs.job_data, '$[1].order')) = <linked-workflow-step-order>
ORDER BY
  background_jobs.created_at,
  background_jobs.id;
```

Record:

- **Background job ids**
- **Job created dates**
- **Step ids represented by the same hash/order**
- **Whether the step id changed over time**
- **Serialized performer id, if present**

If this query returns rows, the email was queued. If the status is `completed`, the job processor completed the mailer job.

## Step 4: Check Current Step Rows for the Candidate Step Ids

Use the step ids found in Step 3.

```sql
SELECT
  workflows.id AS workflow_id,
  workflows.slug,
  workflows.title AS workflow_title,
  workflow_steps.id AS workflow_step_id,
  workflow_steps.[order] AS workflow_step_order,
  workflow_steps.title AS workflow_step_title,
  workflow_steps.status AS workflow_step_status,
  workflow_steps.is_next_step,
  workflow_steps.performer_id,
  performer_users.email AS performer_email,
  performer_users.display_name AS performer_display_name,
  workflow_steps.action_id,
  workflow_actions.title AS workflow_action_title,
  workflow_steps.notes,
  workflow_steps.actual_start_date,
  workflow_steps.actual_complete_date,
  workflow_steps.created_at,
  workflow_steps.updated_at,
  workflow_steps.deleted_at
FROM
  workflows
  INNER JOIN workflow_steps
    ON workflow_steps.workflow_id = workflows.id
  LEFT JOIN workflow_actions
    ON workflow_actions.id = workflow_steps.action_id
  LEFT JOIN users AS performer_users
    ON performer_users.id = workflow_steps.performer_id
WHERE
  workflows.id = <workflow-id>
  AND workflow_steps.id IN (<workflow-step-id-1>, <workflow-step-id-2>)
ORDER BY
  workflow_steps.created_at,
  workflow_steps.id;
```

Interpretation rules:

- **Current `workflow_steps.[order]` may differ from the serialized job order.** Use the job row for what the email linked to when queued.
- **Current row state helps explain later outcome.** It may show completion, performer, or notes added after the email.

## Step 7: Reconstruct Access for Candidate Step Ids

Use this when the question is “why would this person receive or see this step?” Include deleted access rows when reconstructing history.

```sql
SELECT
  workflows.slug,
  workflows.title AS workflow_title,
  workflow_steps.id AS workflow_step_id,
  workflow_steps.[order] AS workflow_step_order,
  workflow_steps.title AS workflow_step_title,
  workflow_actions.title AS workflow_action_title,
  workflow_step_players.id AS workflow_step_player_id,
  CASE
    WHEN workflow_step_players.user_id IS NOT NULL THEN 'direct user player'
    WHEN workflow_step_players.position_id IS NOT NULL THEN 'position player'
    WHEN workflow_step_players.team_id IS NOT NULL THEN 'team player'
    ELSE 'unknown player source'
  END AS workflow_step_player_type,
  workflow_step_players.user_id AS player_user_id,
  player_users.email AS player_user_email,
  player_users.display_name AS player_user_display_name,
  workflow_step_players.position_id AS player_position_id,
  player_positions.title AS player_position_title,
  workflow_step_players.team_id AS player_team_id,
  player_teams.name AS player_team_name,
  workflow_step_access_by_users.id AS workflow_step_access_by_user_id,
  workflow_step_access_by_users.user_id AS access_user_id,
  access_users.email AS access_user_email,
  access_users.display_name AS access_user_display_name,
  workflow_step_access_by_users.user_position_id,
  user_positions.role AS user_position_role,
  user_positions.start_date AS user_position_start_date,
  user_positions.end_date AS user_position_end_date,
  workflow_step_access_by_users.user_team_id,
  user_teams.sources AS user_team_sources,
  workflow_step_access_by_users.position_team_id,
  workflow_step_access_by_users.delegation_id,
  delegations.delegator_id,
  delegator_users.email AS delegator_email,
  delegator_users.display_name AS delegator_display_name,
  delegations.delegatee_id,
  delegatee_users.email AS delegatee_email,
  delegatee_users.display_name AS delegatee_display_name,
  delegations.start_date AS delegation_start_date,
  delegations.end_date AS delegation_end_date,
  workflow_step_access_by_users.created_at AS access_created_at,
  workflow_step_access_by_users.updated_at AS access_updated_at,
  workflow_step_access_by_users.deleted_at AS access_deleted_at
FROM
  workflow_step_access_by_users
  INNER JOIN workflows
    ON workflows.id = workflow_step_access_by_users.workflow_id
  INNER JOIN workflow_steps
    ON workflow_steps.id = workflow_step_access_by_users.workflow_step_id
  LEFT JOIN workflow_actions
    ON workflow_actions.id = workflow_steps.action_id
  INNER JOIN workflow_step_players
    ON workflow_step_players.id = workflow_step_access_by_users.workflow_step_player_id
  LEFT JOIN users AS player_users
    ON player_users.id = workflow_step_players.user_id
  LEFT JOIN positions AS player_positions
    ON player_positions.id = workflow_step_players.position_id
  LEFT JOIN teams AS player_teams
    ON player_teams.id = workflow_step_players.team_id
  INNER JOIN users AS access_users
    ON access_users.id = workflow_step_access_by_users.user_id
  LEFT JOIN user_positions
    ON user_positions.id = workflow_step_access_by_users.user_position_id
  LEFT JOIN user_teams
    ON user_teams.id = workflow_step_access_by_users.user_team_id
  LEFT JOIN delegations
    ON delegations.id = workflow_step_access_by_users.delegation_id
  LEFT JOIN users AS delegator_users
    ON delegator_users.id = delegations.delegator_id
  LEFT JOIN users AS delegatee_users
    ON delegatee_users.id = delegations.delegatee_id
WHERE
  workflow_step_access_by_users.workflow_id = <workflow-id>
  AND workflow_step_access_by_users.workflow_step_id IN (<workflow-step-id-1>, <workflow-step-id-2>)
ORDER BY
  workflow_steps.[order],
  workflow_steps.id,
  workflow_step_players.id,
  access_users.display_name,
  workflow_step_access_by_users.id;
```

Interpretation rules:

- **Direct user player:** the player user is the access user.
- **Position player:** access comes from `user_position_id` and role/date state.
- **Team player:** access comes from `user_team_id` or `position_team_id`.
- **Delegation:** `delegation_id` shows delegated access. Compare delegation dates to job dates.
- **Deleted access rows:** a deleted row may still be relevant if it was active at job time.

## Step 8: Explain “to Unblock Process for <User>”

The mailer wording uses the current step action plus the next later step action and players. Use this query to find the next `Process` step after each Review candidate.

```sql
SELECT
  review_workflow_steps.id AS review_workflow_step_id,
  review_workflow_steps.[order] AS review_workflow_step_order,
  review_workflow_steps.title AS review_workflow_step_title,
  process_workflow_steps.id AS process_workflow_step_id,
  process_workflow_steps.[order] AS process_workflow_step_order,
  process_workflow_steps.title AS process_workflow_step_title,
  process_workflow_actions.title AS process_workflow_action_title,
  workflow_step_players.id AS process_workflow_step_player_id,
  workflow_step_players.user_id AS process_player_user_id,
  player_users.email AS process_player_email,
  player_users.display_name AS process_player_display_name,
  workflow_step_players.position_id AS process_player_position_id,
  positions.title AS process_player_position_title,
  workflow_step_players.team_id AS process_player_team_id,
  teams.name AS process_player_team_name,
  workflow_step_players.created_at AS process_player_created_at,
  workflow_step_players.deleted_at AS process_player_deleted_at
FROM
  workflow_steps AS review_workflow_steps
  CROSS APPLY (
    SELECT TOP 1
      process_candidate_workflow_steps.*
    FROM
      workflow_steps AS process_candidate_workflow_steps
      LEFT JOIN workflow_actions AS process_candidate_workflow_actions
        ON process_candidate_workflow_actions.id = process_candidate_workflow_steps.action_id
    WHERE
      process_candidate_workflow_steps.workflow_id = review_workflow_steps.workflow_id
      AND process_candidate_workflow_steps.[order] > review_workflow_steps.[order]
      AND (
        process_candidate_workflow_steps.title = 'Process'
        OR process_candidate_workflow_actions.title = 'Process'
      )
    ORDER BY
      process_candidate_workflow_steps.[order],
      process_candidate_workflow_steps.id
  ) AS process_workflow_steps
  LEFT JOIN workflow_actions AS process_workflow_actions
    ON process_workflow_actions.id = process_workflow_steps.action_id
  LEFT JOIN workflow_step_players
    ON workflow_step_players.workflow_step_id = process_workflow_steps.id
  LEFT JOIN users AS player_users
    ON player_users.id = workflow_step_players.user_id
  LEFT JOIN positions
    ON positions.id = workflow_step_players.position_id
  LEFT JOIN teams
    ON teams.id = workflow_step_players.team_id
WHERE
  review_workflow_steps.workflow_id = <workflow-id>
  AND review_workflow_steps.id IN (<review-workflow-step-id-1>, <review-workflow-step-id-2>)
ORDER BY
  review_workflow_steps.id,
  process_workflow_steps.[order],
  workflow_step_players.id;
```

Use this to answer:

- **Why does the email mention `Review`?** The linked step action/title is Review.
- **Why does it mention `Process`?** The next later process step action/title is Process.
- **Why does it mention a person?** That person was a player on the next process step at the relevant time.

## Step 10: Check Delegations Active on Mail Dates

Use this when access could come through delegation. Start with investigation user ids and the organization id.

```sql
WITH step_order_mailer_jobs AS (
  SELECT
    background_jobs.id AS background_job_id,
    CAST(background_jobs.created_at AS date) AS job_created_date
  FROM
    background_jobs
  WHERE
    background_jobs.queue_name = 'mailer'
    AND background_jobs.job_name = 'NextStepMailer'
    AND JSON_VALUE(background_jobs.job_data, '$[0].slug') = '<workflow-slug>'
    AND TRY_CONVERT(int, JSON_VALUE(background_jobs.job_data, '$[1].order')) = <linked-workflow-step-order>
)
SELECT
  step_order_mailer_jobs.background_job_id,
  step_order_mailer_jobs.job_created_date,
  delegations.id AS delegation_id,
  delegations.delegator_id,
  delegator_users.email AS delegator_email,
  delegator_users.display_name AS delegator_display_name,
  delegations.delegatee_id,
  delegatee_users.email AS delegatee_email,
  delegatee_users.display_name AS delegatee_display_name,
  delegations.start_date,
  delegations.end_date,
  delegations.deleted_at
FROM
  step_order_mailer_jobs
  INNER JOIN delegations
    ON delegations.start_date <= step_order_mailer_jobs.job_created_date
    AND (
      delegations.end_date IS NULL
      OR delegations.end_date >= step_order_mailer_jobs.job_created_date
    )
  INNER JOIN users AS delegator_users
    ON delegator_users.id = delegations.delegator_id
  INNER JOIN users AS delegatee_users
    ON delegatee_users.id = delegations.delegatee_id
WHERE
  delegations.organization_id = <organization-id>
  AND delegations.deleted_at IS NULL
  AND (
    delegations.delegator_id IN (<investigation-user-id-1>, <investigation-user-id-2>)
    OR delegations.delegatee_id IN (<investigation-user-id-1>, <investigation-user-id-2>)
  )
ORDER BY
  step_order_mailer_jobs.background_job_id,
  delegations.delegator_id,
  delegations.delegatee_id;
```

If this returns empty, delegation is probably not the reason for the specific mailer event.

## Step 12: Build the Explanation Logic

Before writing the final report, reduce the evidence into a small set of claims.

Use this structure:

- **Mailer source:** Identify the `NextStepMailer` job rows and statuses.
- **Link mapping:** State that `#step-<order>` mapped to `workflow_steps.[order] = <order>`.
- **Step identity over time:** Identify whether the same hash/order mapped to one step id or multiple step ids across jobs.
- **Trigger events:** Identify the history events nearest each job, such as completion, send-back, reorder, player add, or player remove.
- **Recipient/access path:** Explain whether access came from a direct user player, position player, team player, or delegation.
- **Unblock wording:** Explain the current step action and the next later process/action step used in the wording.
- **Excluded paths:** State whether `email_histories` or delegation did not explain the reported mailer.

If the evidence shows repeated jobs for the same hash/order, check whether each job is explained by one of these common patterns:

- **A step became next:** A previous step completed and exposed the linked step.
- **A workflow was sent back:** A prior step became next again.
- **A replacement step was inserted:** A new step moved into the linked order.
- **Players changed:** A player add/remove caused new access and a new notification.
- **Steps were reordered:** The visible hash stayed the same while the underlying step id changed.

## Step 13: Write the Shareable Report

Write the report as a narrative, not a query dump. The report should let a reader answer:

- **Was the email expected system behavior?**
- **Which step did the email link point to?**

## Report Formatting Patterns

- **Start with the answer:** Put the conclusion before the timeline.
- **Separate hash meaning from step identity:** Always state that `#step-<order>` means step order, not step id.
- **Group by event phase:** Use headings like `May 4`, `May 6: workflow sent back`, or `After player changes`.
- **Use bold actors and roles:** Bold the person, role, position, or system component doing the action.
- **Use short bullets:** Prefer one event per bullet.
- **Avoid raw query output:** Summarize rows into facts instead of pasting result tables.
- **Avoid unnecessary ids:** Include step ids and job ids only when they disambiguate repeated order values.
- **Call out uncertainty:** Use `could have`, `appears to`, or `inconclusive` when evidence is indirect.
- **Distinguish recipient from access:** If SMTP recipient rows are unavailable, say access is reconstructed from step access, not direct SMTP acceptance.
- **End with user experience:** If behavior is technically valid but confusing, explain what made it confusing.

## Privacy and Sanitization Checklist

Before sharing the report outside the investigation context:

- **Remove email addresses:** Keep display names or roles only if necessary.
- **Remove raw SQL results:** Do not include pasted result rows or internal comments.
- **Minimize personal names:** Prefer role labels unless a specific actor is required for the explanation.
- **Minimize exact timestamps:** Use dates or approximate sequence unless exact timing is necessary.
- **Remove internal notes text:** Paraphrase workflow notes unless the exact wording is required.
- **Review workflow titles and slugs:** Treat them as potentially sensitive business context.
- **Keep ids only when useful:** Step ids can be useful to explain hash/order ambiguity; remove unrelated user ids and job ids.

## Common Pitfalls

- `#step-<order>` can point at one step id in queued jobs and a different step id later if steps are reordered.
- `email_histories` can be empty for the reported email while `background_jobs` contains completed `NextStepMailer` rows.
- Workflow history often explains the confusing sequence: completion, revert, send-back, replacement step creation, player changes, and reorder events.
- Active-only access queries can miss the historical explanation; deleted step player and access rows may be necessary evidence.
- Delegation can be noisy globally, so check it against job dates and candidate users rather than using it as the starting point.
